# Code of Conduct

## Purpose
Set clear behavioral standards for contributors, maintainers, and users of this FastAPI + React Native project. Keep collaboration productive and respectful.

## Scope
Applies to all interactions in issue trackers, pull requests, code reviews, commits, chat channels, mailing lists, in-person meetings, and any public representation of the project.

## Our Standards
Be professional and direct.

- Use clear, concise language.  
- Prioritize facts and reproducible details in technical discussion.  
- Treat people with respect regardless of experience, role, or background.  
- Accept and give constructive feedback. Focus on code and behavior, not personality.  
- Follow security best practices when discussing vulnerabilities. Use private channels for disclosure.  

## Unacceptable Behavior
Do not engage in:

- Harassment, threats, intimidation, or stalking.  
- Abusive, demeaning, or discriminatory language or actions targeting gender, race, religion, disability, sexual orientation, age, nationality, or any protected characteristic.  
- Personal attacks, doxxing, or public shaming.  
- Offensive images, slurs, or derogatory jokes.  
- Repeated, disruptive behavior after a request to stop.  
- Posting sensitive data (secrets, credentials, PII) in public spaces.  

## Technical Etiquette
- Open issues with a clear title, minimal reproduction steps, expected vs actual behavior, logs, and environment (FastAPI/uvicorn version, Python version, React Native version, device or emulator).  
- Submit pull requests for a single purpose. Include concise description, related issue, test plan, and migration notes if any.  
- Add tests for logic changes. For API changes, include OpenAPI schema updates and migration guidance.  
- Respect code style. If no formatter is enforced, explain style changes in PRs.  
- Use feature branches. Rebase or merge regularly to avoid massive conflicts.  
- Label breaking changes explicitly. Provide upgrade steps and deprecation timeline.  

## Security and Vulnerability Reporting
- Do not publish security issues publicly.  
- Report vulnerabilities privately to the maintainers or security contact below.  
- Provide reproduction steps, affected versions, and recommended mitigation if possible.  
- Maintainers will acknowledge receipt within 48 hours and provide a remediation timeline.  

## Enforcement and Consequences
- Reports will be reviewed by maintainers confidentially.  
- Possible actions: private warning, temporary suspension from channels, removal of commit access, public notice, or project ban.  
- Severe or repeated violations may result in removal of contribution privileges and blocking from project infrastructure.  

## Reporting Procedure
Prefer private channels: email [security or maintainer contact] or create a private issue if platform supports.

Provide:
- Your contact details  
- Description of incident  
- Names/handles involved  
- Time, place, and evidence (logs, links, screenshots)  

If you are uncomfortable contacting maintainers, notify the project host (e.g., organization admin) or use platform abuse channels.

## Conflict Resolution
- Maintainers mediate disputes. They may consult neutral community members.  
- Decisions aim for transparency. If you disagree with a decision, file a formal appeal to the maintainers with new evidence.  

## Contribution and Community
- Contributions are welcome. Follow this Code of Conduct and the `CONTRIBUTING.md` in the repo.  
- Mentoring and onboarding are encouraged. Tag issues with `good first issue` or `help wanted`.  

## Licensing and Attribution
- Contributions are accepted under the project license. See `LICENSE` file.  
- By contributing you agree your work will be licensed per the repo terms.  

## Maintainers’ Responsibilities
- Enforce this Code of Conduct impartially.  
- Maintain clear channels for reporting.  
- Keep community interactions professional and timely.
